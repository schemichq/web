Schemic lets you describe a PostgreSQL table once, in Zod, and derive three things from that single definition: the SQL schema (**DDL**), runtime **validation**, and a fully-typed **JS-to-database mapping**. There is no code generation step and no separate schema language. Your Zod schema *is* the schema.

You write this:

<CodeBlock title="database/schema/tables.ts" lang="ts">

```ts
import { defineTable, s, sqlExpr } from "@schemic/postgres";

export const user = defineTable("user", {
  email: s.varchar(255).$unique(),
  name: s.text(),
  age: s.smallint().optional(),
  createdAt: s.timestamptz().$default(sqlExpr("now()")),
});
```

</CodeBlock>

…and the same definition gives you the DDL below, a validator, and an `App` type whose `createdAt` is a JavaScript `Date` even though Postgres stores it as `timestamptz`.

<CodeBlock title="Generated DDL" lang="postgresql">

```sql
CREATE TABLE "user" (
  "id" text PRIMARY KEY,
  "age" smallint,
  "createdAt" timestamp with time zone NOT NULL DEFAULT now(),
  "email" varchar(255) NOT NULL,
  "name" text NOT NULL
);
CREATE UNIQUE INDEX "user_email_key" ON "user" ("email");
```

</CodeBlock>

You never wrote the `id` primary key — Schemic adds an implicit `"id" text PRIMARY KEY` unless you declare your own.

## What you get from one definition

<PillarGrid>
  <PillarCard icon="database" title="Postgres DDL">
    Emit `CREATE TABLE`, columns, primary and foreign keys, unique and secondary indexes, defaults, and checks straight from your schema. No DDL to hand-write or keep in sync.
  </PillarCard>
  <PillarCard icon="shield-check" title="Runtime validation">
    Every field is a real Zod schema — the `z.*` API you already know — so parsing, refinements, formats, and error messages all behave exactly as in Zod.
  </PillarCard>
  <PillarCard icon="layers" title="Typed JS-to-DB mapping">
    Read and write rows through codecs that convert between your app values and the Postgres wire format, with static types for both sides.
  </PillarCard>
</PillarGrid>

A fourth thing falls out of the same source of truth: a declarative, reviewable [migration history](/docs/concepts/migrations-model) the CLI generates by diffing your schema against the database.

<Callout variant="note" title="Standard Schema compatible">
Every `s.*` field implements [Standard Schema](https://standardschema.dev) — the shared interface for TypeScript validation libraries — so you can hand a Schemic field straight to anything that speaks it (tRPC, TanStack Form and Router, and the rest), no adapter needed.
</Callout>

## The mental model

A field is an ordinary Zod schema plus a little Postgres metadata. The mapping between your code and the database rides Zod's two native channels:

- The **app side** (`z.output`) is the value you work with in TypeScript: a `Date`, a `string`, a `Uint8Array`.
- The **wire side** (`z.input`) is what Postgres stores and returns.

A codec moves a value between the two: `decode` reads a row into app values, and `encode` writes app values back. That is why a `timestamptz` column is a `Date` in your code, for free. Keeping these two sides distinct is the core idea; the [encoded and decoded sides](/docs/concepts/app-and-wire) guide covers it in full.

## Prerequisites

<CompatBadges items={["PostgreSQL via PGlite (embedded)", "Zod 4", "TypeScript 5.x", "Node 18+ or Bun"]} />

<Callout variant="note" title="Alpha — PGlite today">
The Postgres driver is `0.1.0-alpha`. Today it runs on embedded [PGlite](https://pglite.dev) — a real Postgres compiled to WASM — so authoring, DDL emit, introspection, and migrations all round-trip with **no database server to install**. Coverage spans tables, columns and native types, indexes (including gin/gist/brin/hash and partial), foreign keys (including composite and non-`id`), enums, sequences, domains, views, materialized views, functions, triggers, and RLS policies. Network Postgres (node-postgres) and multiple schemas are on the beta roadmap. See the [coverage map](https://github.com/schemichq/schemic/blob/main/drivers/postgres/docs/COVERAGE.md) for the feature-by-feature status, and pin a version.
</Callout>
